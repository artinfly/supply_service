## Архитектура проекта

```
Браузер (HTML + JavaScript)
         ↕  HTTP запросы
    Django (Python) — логика
         ↕  SQL запросы
    PostgreSQL — база данных
```

Пользователь открывает страницу в браузере. Django принимает запрос, выполняет SQL-запросы к PostgreSQL и возвращает либо HTML-страницу, либо JSON-данные. JavaScript на странице получает JSON и рисует таблицы без перезагрузки страницы.

## База данных

### Таблицы приложения

#### `igk_stat_data` — основная таблица договоров

Каждая строка — одна позиция договора (один этап, один тип платежа).

| Поле | Тип | Описание |
|------|-----|----------|
| `pp_id` | SERIAL | Уникальный номер записи (автоинкремент) |
| `igk` | VARCHAR | Номер государственного контракта (ГК) |
| `c_agent` | VARCHAR | Контрагент (поставщик) |
| `cfo` | VARCHAR | Центр финансовой ответственности |
| `contract` | VARCHAR | Номер договора |
| `status` | VARCHAR | Состояние договора |
| `payment_type` | VARCHAR | Тип платежа: Аванс или Постоплата |
| `item` | VARCHAR | Предмет договора |
| `order` | VARCHAR | Номер заказа |
| `plan` | FLOAT | Плановая сумма, руб. |
| `fact` | FLOAT | Фактически оплачено, руб. |
| `tolerance` | FLOAT | Допуск, % |
| `stage` | VARCHAR | Этап графика платежей |
| `y25` | BOOLEAN | Относится к 2025 году |
| `y26` | BOOLEAN | Относится к 2026 году |
| `y27` | BOOLEAN | Относится к 2027 году |
| `is_deleted` | BOOLEAN | Признак удалённой записи |
| `plan_date` | VARCHAR | Дата платежа по плану |
| `c_date` | VARCHAR | Дата создания договора |

#### `contracts_history` — история изменений

Фиксирует каждое изменение статуса, плана или факта при повторной загрузке Excel.

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | SERIAL | Уникальный номер |
| `hash` | BYTEA | MD5-хеш позиции договора (ключ привязки) |
| `old_status` | VARCHAR | Статус до изменения |
| `new_status` | VARCHAR | Статус после изменения |
| `update_date` | DATE | Дата изменения |
| `upload_date` | DATE | Дата загрузки файла |
| `old_plan` | NUMERIC | План до изменения |
| `new_plan` | NUMERIC | План после изменения |
| `old_fact` | NUMERIC | Факт до изменения |
| `new_fact` | NUMERIC | Факт после изменения |
| `plan_changed_date` | DATE | Дата изменения плана |
| `fact_changed_date` | DATE | Дата изменения факта |

#### `staging_excel` — промежуточная таблица

Временное хранилище сырых данных из Excel до нормализации. Все поля — строки (VARCHAR), типы ещё не преобразованы.

#### `nsi_igk` и `nsi_cfo` — справочники

Списки уникальных значений ИГК и ЦФО. Заполняются автоматически при нормализации.

### `settings.py` — настройки

Центральный файл конфигурации Django. Все чувствительные данные (пароль БД, секретный ключ) читаются из переменных окружения через `.env` файл.

Ключевые параметры:
- `SECRET_KEY` — секретный ключ шифрования сессий
- `DEBUG` — режим отладки (True на разработке, False на сервере)
- `ALLOWED_HOSTS` — разрешённые адреса для обращения к серверу
- `DATABASES` — параметры подключения к PostgreSQL
- `YEARS = [2025, 2026, 2027]` — список активных годов *(см. раздел "Добавление нового года")*

### `import_excel.py` — загрузка Excel

Django management command. Запускается командой:
```bash
python manage.py import_excel путь/к/файлу.xlsx
```

Что делает:
1. Открывает xlsx через openpyxl в режиме только чтение
2. Читает первую строку как заголовки
3. Сопоставляет русские названия колонок с полями таблицы через `COLUMN_MAP`
4. Очищает таблицу `staging_excel` (`TRUNCATE`)
5. Вставляет все строки из Excel в `staging_excel`

Поддерживаемые колонки Excel:

| Колонка в Excel | Поле в БД |
|----------------|-----------|
| ИГК | igk |
| Контрагент | kontragent |
| ЦФО | cfo |
| Договор | dogovor |
| Состояние | sostoyanie |
| Тип платежа | tip_platezha |
| Предмет | predmet |
| Заказ | zakaz |
| ПЛАН | plan |
| ФАКТ | fakt |
| Тол | tol |
| Этап графика | etap_grafika |
| ДатаПЛАН | dataplan |
| Создан | sozdan |

### `normalize_staging.py` — нормализация данных

Django management command. Запускается командой:
```bash
python manage.py normalize_staging
```

Что делает:
1. Добавляет новые ЦФО и ИГК в справочники
2. Читает все строки из `staging_excel`
3. Преобразует типы: строки → числа, определяет флаги года (`y25`, `y26`, `y27`) по полю `ДатаПЛАН`
4. Сравнивает с текущими данными в `igk_stat_data` — если статус, план или факт изменились, записывает в `contracts_history`
5. Очищает `igk_stat_data` и вставляет нормализованные данные

**Логика определения года** — берутся первые 4 символа поля `ДатаПЛАН`:
- `2025-...` → `y25 = TRUE`
- `2026-...` → `y26 = TRUE`
- `2027-...` → `y27 = TRUE`

### `setup_groups.py` — создание групп

Django management command. Запускается один раз при первоначальной настройке:
```bash
python manage.py setup_groups
```

Создаёт две группы пользователей: `viewer` и `operator`.

## Развёртка на Windows

- Python 3.13
- PostgreSQL 16+

```bash
# 1. Перейти в папку
cd supply_service

# 2. Создать виртуальное окружение
python -m venv .venv

pip install --no-index --find-links=wheels -r requirements.txt
```

Создать файл `.env` в корне проекта:
```
DJANGO_SECRET_KEY=любая-длинная-случайная-строка-минимум-50-символов
DJANGO_DEBUG=True
DJANGO_ALLOWED_HOSTS=localhost,127.0.0.1
DB_NAME=supply_service
DB_USER=postgres
DB_PASSWORD=пароль_от_postgresql
DB_HOST=localhost
DB_PORT=5432
```

### Создание базы данных

1. Открыть pgAdmin
2. Создать базу данных `supply_service` (если не существует)
3. Открыть Query Tool для этой базы
4. Выполнить содержимое файла `setup_db.sql`

```bash
python manage.py migrate          # системные таблицы Django
python manage.py setup_groups     # группы пользователей
python manage.py createsuperuser  # администратор
python manage.py collectstatic    # статические файлы
python manage.py runserver        # запуск сервера
```

Открыть в браузере: **http://127.0.0.1:8000**

## Добавление нового года

Когда наступает новый год (например 2028), нужно внести изменения в **4 места**:

### 1. `reports/views.py`

Найти константу `YEARS` в начале файла:
```python
YEARS = [2025, 2026, 2027]
```
Добавить новый год:
```python
YEARS = [2025, 2026, 2027, 2028]
```

`YEAR_COL` обновлять не нужно — он строится автоматически из `YEARS`.

### 2. `reports/management/commands/normalize_staging.py`

Найти функцию `year_flags`:
```python
def year_flags(dataplan):
    try:
        y = int(str(dataplan).strip()[:4])
        return y == 2025, y == 2026, y == 2027
    except Exception:
        return False, False, False
```
Добавить новый год:
```python
def year_flags(dataplan):
    try:
        y = int(str(dataplan).strip()[:4])
        return y == 2025, y == 2026, y == 2027, y == 2028
    except Exception:
        return False, False, False, False
```

### 3. База данных — добавить колонку

Выполнить в pgAdmin (Query Tool для базы `supply_service`):
```sql
ALTER TABLE igk_stat_data ADD COLUMN y28 BOOLEAN DEFAULT FALSE;
ALTER TABLE staging_excel ADD COLUMN ... -- не нужно, staging не хранит флаги
```

### 4. `reports/models.py`

Добавить поле в модель `IgkStatData`:
```python
y28 = models.BooleanField(null=True)
```

### 5. `normalize_staging.py` — INSERT

Найти `new_data.append(...)` и добавить `y28` в кортеж, а также добавить в INSERT SQL.

> **Важно:** после всех изменений перезапустить сервер и повторно загрузить Excel-файл командами `import_excel` и `normalize_staging`.

### Что происходит при повторной загрузке

При каждой загрузке нового Excel-файла система сравнивает данные с предыдущей версией. Если для какой-либо позиции изменился статус, плановая сумма или фактическая оплата — изменение фиксируется в таблице `contracts_history` и отображается в разделах **История изменений**.

## Структура Excel-файла для загрузки

Файл должен быть формата `.xlsx`. Первая строка — заголовки колонок. Порядок колонок не важен — система определяет их по названию.

**Обязательные колонки** (названия должны совпадать точно):

```
ИГК | Контрагент | ЦФО | Договор | Состояние | Тип платежа |
Предмет | Заказ | ПЛАН | ФАКТ | Тол | Этап графика | ДатаПЛАН | Создан
```

**Формат поля ДатаПЛАН:** `ГГГГ-ММ-ДД` или любой формат начинающийся с 4-значного года. По первым 4 символам система определяет к какому году относится позиция.

**Типы платежей:** система различает `Аванс` и `Постоплата`. Все остальные значения игнорируются при расчёте авансовых показателей.

**Статусы договоров:**
- Заключённые: `Исполняется`, `Возвращен на уточнение`, `На согласовании`, `Подписан`, `На утверждении`
- Незаключённые: `Черновик`
- Расторгнутые: `Расторгнут`
