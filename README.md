## Архитектура проекта

```
Браузер (HTML + JavaScript)
         ↕  HTTP / AJAX
    Django (Python) — логика
         ↕  SQL запросы
    PostgreSQL — хранение данных
```

Пользователь открывает страницу в браузере. Django принимает запрос, выполняет SQL-запросы к PostgreSQL и возвращает HTML-страницу. Таблицы загружаются асинхронно — JavaScript получает JSON от API-эндпоинтов и рисует таблицы без перезагрузки страницы.

Для сложных аналитических запросов (агрегации с фильтрацией по статусам и годам, PostgreSQL-специфичные конструкции FILTER и NULLIF) используется сырой SQL через Django DB API. ORM применяется для описания схемы данных и миграций.

## База данных

Схема БД описана в `reports/models.py` через Django ORM. Миграции создаются командой `makemigrations` и применяются командой `migrate`. При развёртывании с нуля таблицы создаются автоматически.

### `igk_stat_data` — основная таблица договоров

Каждая строка — одна позиция договора (один этап, один тип платежа).

| Поле | Тип | Описание |
|------|-----|----------|
| pp_id | SERIAL PK | Уникальный номер записи |
| igk | VARCHAR(500) | Номер государственного контракта |
| c_agent | VARCHAR(500) | Контрагент (поставщик) |
| cfo | VARCHAR(500) | Центр финансовой ответственности |
| contract | VARCHAR(500) | Номер договора |
| status | VARCHAR(500) | Состояние договора |
| payment_type | VARCHAR(500) | Тип платежа: Аванс или Постоплата |
| item | VARCHAR(500) | Предмет договора |
| order | VARCHAR(500) | Номер заказа |
| plan | FLOAT | Плановая сумма, руб. |
| fact | FLOAT | Фактически оплачено, руб. |
| tolerance | FLOAT | Допуск, % |
| stage | VARCHAR(250) | Этап графика платежей |
| y25 | BOOLEAN | Относится к 2025 году |
| y26 | BOOLEAN | Относится к 2026 году |
| y27 | BOOLEAN | Относится к 2027 году |
| is_deleted | BOOLEAN | Признак удалённой записи |
| plan_date | VARCHAR(50) | Дата платежа по плану |
| c_date | VARCHAR(256) | Дата создания записи |

### `contract_counts_snapshot` — снимки количества договоров

Записывается при каждой загрузке данных. Используется для расчёта дельты (сколько договоров заключено за период) в отчёте КДР.

| Поле | Тип | Описание |
|------|-----|----------|
| id | SERIAL PK | Уникальный номер |
| upload_date | DATE | Дата загрузки файла |
| igk | VARCHAR(10) | Последние 4 символа ИГК |
| cfo | VARCHAR(10) | ЦФО |
| year_col | VARCHAR(5) | Годовой флаг (y25, y26, y27) |
| concluded_count | INTEGER | Количество заключённых договоров |

### `contracts_history` — история изменений

Фиксирует каждое изменение статуса, плана или факта при повторной загрузке Excel. Связь с текущими данными через MD5-хеш ключевых полей.

| Поле | Тип | Описание |
|------|-----|----------|
| id | SERIAL PK | Уникальный номер |
| hash | BYTEA | MD5-хеш позиции договора |
| old_status | VARCHAR(500) | Статус до изменения |
| new_status | VARCHAR(500) | Статус после изменения |
| update_date | DATE | Дата изменения |
| upload_date | DATE | Дата загрузки файла |
| old_plan | FLOAT | План до изменения |
| new_plan | FLOAT | План после изменения |
| old_fact | FLOAT | Факт до изменения |
| new_fact | FLOAT | Факт после изменения |
| plan_changed_date | DATE | Дата изменения плана |
| fact_changed_date | DATE | Дата изменения факта |

### `staging_excel` — промежуточная таблица

Временное хранилище сырых данных из Excel до нормализации. Все поля — текстовые, типы преобразуются при нормализации.

### `nsi_igk` и `nsi_cfo` — справочники

Списки уникальных ИГК и ЦФО. Заполняются автоматически при нормализации.

## Модули приложения

### `views/pages.py` — страницы

Функции, которые возвращают HTML-страницы: главная, таблицы, загрузка, формирование отчётов.

### `views/api.py` — API

Функции, которые возвращают JSON-данные для JavaScript-таблиц. Вызывают SQL из `services/queries.py`.

### `views/exports.py` — экспорт

Функции формирования xlsx-файлов для скачивания. Используют `services/excel.py` для простых отчётов и `services/pivot.py` для отчёта ИГК со сводной таблицей.

### `services/queries.py` — SQL-запросы

Все SQL-запросы вынесены в один файл как функции, возвращающие строки. Это позволяет менять запросы в одном месте, не трогая views.

### `services/excel.py` — генерация xlsx

Общие функции: `make_wb()` создаёт xlsx с форматированием заголовков и чисел, `xlsx_response()` оборачивает в HTTP-ответ.

### `services/pivot.py` — сводная таблица

Генерация xlsx на основе шаблона `templateIGK.xlsx` с корректным обновлением pivot-таблицы для любого года.

### `import_excel.py` — загрузка Excel

Читает xlsx, сопоставляет колонки по названиям через `COLUMN_MAP`, записывает в `staging_excel`. Порядок колонок не важен.

### `normalize_staging.py` — нормализация данных

1. Обновляет справочники ЦФО и ИГК
2. Определяет год по полю `ГодИГК` (столбец 39 в Excel)
3. Преобразует типы: строки в числа
4. Сравнивает с текущими данными, фиксирует изменения в `contracts_history`
5. Заменяет данные в `igk_stat_data`
6. Записывает снимки количества договоров в `contract_counts_snapshot`

## Развёртывание

### Требования

- Python 3.13
- PostgreSQL 16+

### Установка

```bash
cd supply_service
python -m venv .venv
.venv\Scripts\activate

pip install --no-index --find-links=wheels -r requirements.txt
```

### Переменные окружения

Создать `.env` в корне:

```
DJANGO_SECRET_KEY=любая-длинная-случайная-строка-минимум-50-символов
DJANGO_DEBUG=True
DJANGO_ALLOWED_HOSTS=localhost,127.0.0.1
DB_NAME=supply_service
DB_USER=postgres
DB_PASSWORD=пароль_от_postgresql
DB_HOST=localhost
DB_PORT=5432
```

```bash
python manage.py migrate
python manage.py setup_groups
python manage.py createsuperuser
python manage.py runserver
```

Открыть: **http://127.0.0.1:8000**

## Добавление нового года

Например, 2028. Изменить 4 места:

**1. `reports/services/queries.py`** — добавить год в константы:

```python
YEARS = [2025, 2026, 2027, 2028]
```

**2. `reports/management/commands/normalize_staging.py`** — добавить в `year_flags` и `YEAR_MAP`:

```python
def year_flags(god_igk):
    try:
        y = int(str(god_igk).strip()[:4])
        return y == 2025, y == 2026, y == 2027, y == 2028
    except Exception:
        return False, False, False, False

YEAR_MAP = [('y25', 2025), ('y26', 2026), ('y27', 2027), ('y28', 2028)]
```

Обновить кортеж в `new_data.append(...)` — добавить `y28` как 14-й элемент, и в INSERT SQL — добавить `%s`.

**3. `reports/models.py`** — добавить поле в `IgkStatData`:

```python
y28 = models.BooleanField(null=True)
```

**4. Создать и применить миграцию:**

```bash
python manage.py makemigrations reports
python manage.py migrate
```

После изменений перезагрузить Excel-файл.

## Загрузка данных

### Через веб-интерфейс

1. Войти как operator или superuser
2. Открыть **Загрузка данных** в сайдбаре
3. Выбрать xlsx-файл и нажать **Загрузить и нормализовать**

### Через командную строку

```bash
python manage.py import_excel путь\к\файлу.xlsx
python manage.py normalize_staging
```

При повторной загрузке система сравнивает данные с предыдущей версией. Изменения статуса, плана или факта фиксируются в `contracts_history` и отображаются в разделе **История изменений**.

## Структура Excel-файла для загрузки

Формат `.xlsx`, первая строка — заголовки. Порядок колонок не важен.

**Обязательные колонки:**

```
ИГК | Контрагент | ЦФО | Договор | Состояние | Тип платежа |
Предмет | Заказ | ПЛАН | ФАКТ | Тол | Этап графика | ДатаПЛАН | Создан | ГодИГК
```

**ГодИГК** (столбец 39) — определяет к какому году относится позиция. Содержит 4-значный год: 2025, 2026 или 2027.

**Типы платежей:** `Аванс` и `Постоплата`. Остальные значения игнорируются при расчёте авансовых показателей.

**Статусы:**

- Заключённые: `Исполняется`, `Возвращен на уточнение`, `На согласовании`, `Подписан`, `На утверждении`
- Незаключённые: `Черновик`
- Расторгнутые: `Расторгнут`
